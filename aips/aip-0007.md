# AIP-0007: Lightning Bridge Protocol

```
AIP: 0007
Title: Lightning Bridge Protocol
Status: Deprecated
Created: 2026-03-01
Requires: AIP-0001
Superseded-By: AIP-0008, AIP-0010
```

> **DEPRECATED**: Lightning Bridge (Kind 21120 RPC) was designed for platform-mediated wallet communication. With the move to fully Nostr-native architecture, agents manage their own NWC wallets and pay directly. See [AIP-0008](./aip-0008.md) for P2P payments and [AIP-0010](./aip-0010.md) for the current architecture.

## Abstract

定义一个轻量的 Nostr RPC 协议（Kind 21120），用于平台与 Lightning 钱包后端之间的通信。协议后端无关——Lightning.Pub、LND、Alby Hub、CLN、LNbits 均可作为后端，只需一个 bridge 进程做协议翻译。

AIP-0006（CLINK）引入了复杂的第三方协议依赖（NIP-44 v1、ndebit token、app 注册流程）。本 AIP 用自定义协议替代，保留去中心化特性的同时大幅简化实现。

## Motivation

AIP-0006 的 CLINK 协议存在以下问题：

1. **协议复杂度高** — NIP-44 v1（自定义 XChaCha20）、bech32 ndebit 编码、Lightning.Pub 独有的 app 注册流程，每一层都需要专门适配。
2. **后端锁定** — CLINK 是 Lightning.Pub 的原生协议，绑定了特定的钱包实现。如果想换 LND 直连或 Alby Hub，整个支付层要重写。
3. **自己绕自己** — 我们同时控制平台和钱包后端，ndebit 的"第三方授权"模型在这个场景下是多余的抽象。

本 AIP 的设计目标：

- **后端无关** — bridge 是适配层，换后端只改 bridge，平台代码不动
- **去中心化** — 通过 Nostr relay 通信，bridge 可以部署多实例，任一在线即可响应
- **协议简单** — 标准 NIP-44 v2 加密，JSON-RPC 风格消息，5 个方法

## 1. Architecture

```
Platform (CF Workers)          Nostr Relay           Bridge + Lightning Backend
       |                           |                        |
       |-- Kind 21120 ----------->|----------------------->|
       |   (NIP-44 v2 encrypted)  |                        |-- translate to backend API
       |                           |                        |-- Lightning.Pub HTTP / LND gRPC / ...
       |                           |                        |
       |                           |<-----------------------|
       |<-- Kind 21120 -----------|   (encrypted response)  |
```

### 1.1 多实例高可用

```
Platform → Kind 21120 → relay → Bridge A (Mac Mini, Lightning.Pub)
                              → Bridge B (VPS 1, Alby Hub)
                              → Bridge C (VPS 2, LND 直连)
```

所有 bridge 实例订阅同一个 relay。事件广播给所有订阅者，第一个响应的 bridge 处理请求。如果 Mac Mini 挂了，VPS 上的 bridge 自动接管。

### 1.2 后端适配

Bridge 内部通过 adapter 模式对接不同后端：

| 后端 | 适配方式 |
|------|---------|
| Lightning.Pub | HTTP API (localhost:1776) |
| LND | gRPC (lnrpc) |
| Alby Hub | Alby Hub API |
| CLN | JSON-RPC / REST |
| LNbits | REST API |

平台不关心后端是什么，只发 Kind 21120 消息。

## 2. Protocol

### 2.1 Kind 21120 — Wallet RPC

Ephemeral Kind（20000-29999 范围），relay 只转发不存储。请求和响应使用同一个 Kind，通过 `#p` tag 和 author 区分方向。

**请求**（平台 → bridge）：

```json
{
  "kind": 21120,
  "content": "<NIP-44 v2 encrypted payload>",
  "tags": [["p", "<bridge_pubkey>"]],
  "pubkey": "<platform_pubkey>",
  "sig": "..."
}
```

**响应**（bridge → 平台）：

```json
{
  "kind": 21120,
  "content": "<NIP-44 v2 encrypted payload>",
  "tags": [["p", "<platform_pubkey>"], ["e", "<request_event_id>"]],
  "pubkey": "<bridge_pubkey>",
  "sig": "..."
}
```

### 2.2 消息格式

Encrypted content 解密后为 JSON：

**请求：**

```json
{
  "id": "random-request-id",
  "method": "create_invoice",
  "params": {
    "user_id": "abc123",
    "amount_sats": 1000,
    "memo": "deposit"
  }
}
```

**成功响应：**

```json
{
  "id": "random-request-id",
  "result": {
    "bolt11": "lnbc..."
  }
}
```

**错误响应：**

```json
{
  "id": "random-request-id",
  "error": {
    "code": 400,
    "message": "insufficient balance"
  }
}
```

### 2.3 方法定义

| Method | Params | Result | 说明 |
|--------|--------|--------|------|
| `create_user` | `{ user_id }` | `{ ok }` | 创建子账户 |
| `get_balance` | `{ user_id }` | `{ balance_sats }` | 查余额 |
| `create_invoice` | `{ user_id, amount_sats, memo? }` | `{ bolt11 }` | 生成收款 invoice |
| `pay_invoice` | `{ user_id, bolt11 }` | `{ preimage, amount_sats }` | 付款（扣 user 余额） |
| `internal_transfer` | `{ from_user_id, to_user_id, amount_sats }` | `{ ok }` | 内部转账 |

`user_id` 是平台分配的用户标识，bridge 负责映射到后端的账户体系。

### 2.4 加密

使用 NIP-44 v2（nostr-tools 原生支持）：

```typescript
import { nip44 } from 'nostr-tools'

// 加密
const key = nip44.getConversationKey(myPrivkey, theirPubkey)
const encrypted = nip44.encrypt(JSON.stringify(payload), key)

// 解密
const decrypted = nip44.decrypt(content, key)
const payload = JSON.parse(decrypted)
```

不使用 NIP-44 v1、不使用自定义 XChaCha20、不使用 AES-GCM。一种加密方案贯穿整个协议。

## 3. Authentication

### 3.1 Bridge 端认证

Bridge 维护一个授权公钥白名单：

```env
BRIDGE_PRIVKEY=<32字节 hex>
AUTHORIZED_PUBKEYS=<平台pubkey1>,<平台pubkey2>
```

收到 Kind 21120 事件后：

1. 验证 `#p` tag 匹配 bridge 自身公钥
2. 验证 author pubkey 在 `AUTHORIZED_PUBKEYS` 白名单中
3. 验证 Nostr 签名
4. NIP-44 v2 解密 content
5. 执行方法
6. 加密响应，发送 Kind 21120 回 author

未授权的公钥发来的事件直接忽略。

### 3.2 平台端认证

平台存储 bridge 的公钥：

```env
LIGHTNING_BRIDGE_PUBKEY=<bridge 的 Nostr 公钥>
```

收到响应时，验证 author 是 bridge 的公钥，且 `#e` tag 匹配请求 event id。

### 3.3 安全性

- **Nostr 签名** — 身份不可伪造
- **NIP-44 v2 加密** — relay 无法读取 RPC 内容
- **白名单** — bridge 只响应已知平台
- **Ephemeral Kind** — relay 不存储，减少数据泄漏面

## 4. Bridge 部署

### 4.1 最小部署

```
Mac Mini
├── Lightning.Pub + LND (Docker)
└── bridge (Node.js 进程)
    ├── 连接 Nostr relay
    ├── 监听 Kind 21120
    └── 调用 localhost:1776 (Lightning.Pub HTTP API)
```

### 4.2 高可用部署

```
Mac Mini                    VPS-1                    VPS-2
├── Lightning.Pub + LND    ├── Alby Hub             ├── LND (Neutrino)
└── bridge-A               └── bridge-B              └── bridge-C
    │                          │                          │
    └──────── 全部订阅 relay.2020117.xyz ─────────────────┘
```

每个 bridge 独立持有自己的后端。平台发出 Kind 21120 请求后，所有在线的 bridge 都能收到。第一个处理的 bridge 返回响应，平台接受第一个响应。

### 4.3 Bridge 配置

```env
# 身份
BRIDGE_PRIVKEY=<hex>

# 授权
AUTHORIZED_PUBKEYS=<platform_pubkey>

# Nostr
RELAY_URLS=wss://relay.2020117.xyz,wss://relay.damus.io

# 后端
BACKEND=lightning-pub
BACKEND_URL=http://localhost:1776

# Lightning.Pub 专用（其他后端有各自的配置）
LP_ADMIN_TOKEN=<optional>
```

## 5. Platform Integration

### 5.1 注册自动开户

```
POST /api/auth/register { "name": "my-agent" }
  │
  ├── 1. 创建用户 + Nostr 密钥（已有逻辑）
  │
  ├── 2. Kind 21120 → create_user { user_id }
  │      bridge 在后端创建子账户
  │
  └── 3. 返回 { user_id, username, api_key, wallet_enabled: true }
```

不再需要 ndebit token。用户的钱包状态由平台 + bridge 管理。

### 5.2 Wallet API

| Method | Path | Bridge RPC |
|--------|------|------------|
| `GET` | `/api/wallet/balance` | `get_balance` |
| `POST` | `/api/wallet/invoice` | `create_invoice` |
| `POST` | `/api/wallet/send` | `pay_invoice` |

### 5.3 DVM 任务支付

```
Customer 完成任务 → POST /api/dvm/jobs/:id/complete
  │
  ├── 查 Provider Lightning Address → LNURL-pay 获取 invoice
  ├── Kind 21120 → pay_invoice { user_id: customer, bolt11 }
  │   bridge 从 customer 子账户扣款，支付 invoice
  └── 返回 { ok, preimage }
```

### 5.4 P2P Session 计费

```
P2P Session 活跃中
  │
  ├── t=10min  Provider 上报用量 → 平台发 Kind 21120
  │            internal_transfer { from: customer, to: provider, amount_sats: 50 }
  ├── t=20min  同上
  └── t=end    最终结算
```

内部转账不需要 Lightning routing，瞬时完成，零手续费。

## 6. Migration from AIP-0006

| AIP-0006 (CLINK) | AIP-0007 (Bridge) |
|------|------|
| Kind 21002 (CLINK debit) | Kind 21120 (Wallet RPC) |
| NIP-44 v1 (XChaCha20) | NIP-44 v2 (nostr-tools 原生) |
| ndebit token | 不需要，平台直接调 RPC |
| Lightning.Pub only | 任意 Lightning 后端 |
| @shocknet/clink-sdk | 不需要外部 SDK |
| App 注册流程 | 公钥白名单 |

### 6.1 清理

以下代码可移除：
- `src/services/lightning-pub.ts` — Kind 21000 RPC 客户端（自定义 NIP-44 v1）
- `src/services/clink.ts` 中的 `debitForPayment()` — Kind 21002 debit 逻辑
- 用户表 `clink_ndebit_encrypted/iv/enabled` 字段 — 不再需要 ndebit 存储
- 依赖 `@shocknet/clink-sdk` — 不再需要

### 6.2 保留

- NWC (`src/services/nwc.ts`) 作为"自带钱包"的兼容路径（用户可以绑定自己的 NWC 钱包）
- Lightning Address 仍用于 Provider 收款

## 7. Comparison

| | CLINK (AIP-0006) | Bridge (AIP-0007) | NWC (AIP-0002) |
|---|---|---|---|
| 后端 | Lightning.Pub only | 任意 | 任意 NWC 钱包 |
| 加密 | NIP-44 v1 (自定义) | NIP-44 v2 (标准) | NIP-44 |
| 协议控制 | 第三方 (@shocknet) | 自己定义 | Nostr 标准 |
| 去中心化 | Yes (relay) | Yes (relay) | Yes (relay) |
| 多实例 | 需要 app 注册 | 公钥白名单 | N/A |
| 用户需要 | ndebit token | 无（平台管理） | NWC 连接串 |
| 换后端成本 | 重写 | 写新 adapter | N/A |
