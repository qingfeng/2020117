# Wallet Auto-Provisioning Implementation Plan

**Goal:** 用户注册时自动通过 Lightning.Pub Nostr RPC 开通钱包，获取 ndebit，实现零配置 agent 支付能力。最终验证：注册新 agent → 充值 → P2P session 支付。

**Architecture:** Platform (CF Workers) 通过 Kind 21000 NIP-44 加密事件与 Lightning.Pub (Mac Mini) 通信，经 relay.2020117.xyz 中转。无需 Lightning.Pub 暴露公网。

**Tech Stack:** nostr-tools (NIP-44 encryption), @noble/curves (secp256k1 ECDH), Cloudflare Workers WebSocket

---

## Phase 1: Lightning.Pub Nostr RPC 客户端

### Task 1: 研究 Lightning.Pub RPC 协议细节

**目标：** 从 Lightning.Pub 源码提取精确的 RPC 协议格式（Kind 21000 event 结构、NIP-44 加密方式、auth token 格式、response sharding）。

**步骤：**
1. 克隆 Lightning.Pub repo，读 `proto/autogenerated/ts/nostr_client.ts` 和 `nostr_transport.ts`
2. 读 ShockWallet `src/Api/nostrHandler.ts` 了解传输层实现
3. 确认：Kind 21000 event tags 结构、content 加密方式（NIP-44 v1 vs v2）、auth token 传递方式
4. 记录所需 RPC 方法的精确 request/response 类型

**产出：** 协议细节文档（追加到此计划文件末尾）

### Task 2: 构建 Nostr RPC 传输层

**文件：** `src/services/lightning-pub.ts`（新建）

**功能：**
- `pubRpc(env, rpcName, body, authLevel)` — 通用 RPC 调用
  - 生成 requestId (uuid)
  - 用平台 Nostr 密钥 + Lightning.Pub pubkey 做 NIP-44 加密
  - 构建 Kind 21000 event，签名
  - 通过 WebSocket 发到 relay，订阅响应
  - 等待匹配 requestId 的响应，解密返回
  - 超时处理（30s）

**依赖：**
- `NOSTR_MASTER_KEY` — 已有，派生平台 Nostr 密钥
- 新环境变量 `LIGHTNING_PUB_PUBKEY` — Lightning.Pub 的 Nostr pubkey
- 新环境变量 `LIGHTNING_PUB_ADMIN_TOKEN` — admin 认证 token（从 nprofile 获取）
- relay 地址从 `NOSTR_RELAYS` 复用

**模式：** 与现有 `src/services/clink.ts` 的 `debitForPayment()` 类似 — WebSocket 连 relay，发事件，等响应。

### Task 3: 实现关键 RPC 方法封装

**文件：** `src/services/lightning-pub.ts`

封装以下方法：

```typescript
// 创建子账户（注册时调用）
async function createWalletAccount(env: Env, identifier: string): Promise<{ ndebit: string }>

// 生成充值 invoice
async function createInvoice(env: Env, userIdentifier: string, amountSats: number): Promise<{ bolt11: string }>

// 查询余额

// 付款（提现）

```

### Task 4: Relay Kind 21000 白名单

**文件：** `relay/src/types.ts`, `relay/src/relay-do.ts`

- 将 Kind 21000 加入 `ALLOWED_KINDS`
- 加入 POW 豁免（和 21002 一样）
- Kind 21000 是 ephemeral (20000-29999)，broadcast 但不存储 — 现有逻辑已处理

## Phase 2: 平台集成

### Task 5: 注册时自动开通钱包

**文件：** `src/routes/api.ts`（修改 `POST /api/auth/register`）

在现有 Nostr 密钥生成后，增加：
1. 调用 `createWalletAccount(env, userId)` 创建 Lightning.Pub 子账户
2. 拿到 ndebit，加密存储到 user 表（复用现有 `encryptNdebit`）
3. 设置 `clink_ndebit_enabled = 1`
4. 返回 `wallet_enabled: true`

**容错：** Lightning.Pub 不可用时注册仍成功，只是没有钱包。用户后续可手动绑定 ndebit。

### Task 6: 钱包 API 端点

**文件：** `src/routes/api.ts`

新增端点：

```
POST /api/wallet/invoice    { amount_sats: number, memo?: string }
  → 返回 { bolt11, amount_sats }

GET /api/wallet/balance
  → 返回 { balance_sats }

POST /api/wallet/send       { bolt11: string }
  → 返回 { preimage, amount_sats }
```

所有端点需要认证，且用户必须有 `clink_ndebit_enabled`。

## Phase 3: Agent 运行时集成

### Task 8: Agent CLI 自动获取 ndebit

**文件：** `worker/src/api.ts`

当 agent 启动时，如果 `.2020117_keys` 中没有 ndebit：
1. 调 `GET /api/me` 检查 `clink_ndebit_enabled`
2. 如果已开通但本地没有 ndebit，调新端点获取（或者注册时返回时就保存了）

实际上 ndebit 由平台加密存储，agent 不需要持有明文 ndebit。P2P session 时通过 `GET /api/me` 拿到 ndebit（需要新增返回字段）。

**重新思考：** Agent 在 P2P session 中需要把 ndebit 发给 provider。两种方式：
- A) 平台返回明文 ndebit 给 agent（安全性较低但简单）
- B) Agent 通过平台 proxy 支付（现有 proxy-debit 模式）

当前 P2P session 已经用方式 A（session_start 时发送 ndebit）。所以注册时需要把 ndebit 原文返回给 agent 保存到 `.2020117_keys`。

### Task 9: 注册流程返回 ndebit

**文件：** `src/routes/api.ts`

`POST /api/auth/register` 返回中增加 `ndebit` 字段（仅在注册响应中返回一次，与 api_key 类似）。

`GET /api/me` 不返回明文 ndebit（安全）。如果 agent 丢失了 ndebit，需要重新生成（新端点 `POST /api/wallet/reissue-ndebit`）。

## Phase 4: 端到端测试

### Task 10: 注册新 agent 并验证钱包

```bash
# 注册
curl -s -X POST https://2020117.xyz/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"name":"test-provider"}' | jq .

# 预期返回：
# { user_id, username, api_key, ndebit: "ndebit1...", wallet_enabled: true }

# 验证钱包
curl -s https://2020117.xyz/api/wallet/balance \
  -H "Authorization: Bearer <api_key>" | jq .
# { balance_sats: 0 }
```

### Task 11: 充值（从已有 agent 转账）

```bash
# test-provider 生成收款 invoice
curl -s -X POST https://2020117.xyz/api/wallet/invoice \
  -H "Authorization: Bearer <test_provider_key>" \
  -H "Content-Type: application/json" \
  -d '{"amount_sats": 500}' | jq .
# { bolt11: "lnbc..." }

# customer-agent 付款
curl -s -X POST https://2020117.xyz/api/wallet/send \
  -H "Authorization: Bearer <customer_key>" \
  -H "Content-Type: application/json" \
  -d '{"bolt11": "lnbc..."}' | jq .
# { preimage: "...", amount_sats: 500 }

# 验证双方余额变化
```

### Task 12: P2P Session 端到端测试

1. 启动 provider agent:
```bash
npx 2020117-agent --agent=test-provider --kind=5100 --processor=none
```

2. 启动 customer session:
```bash
npx -p 2020117-agent 2020117-session --agent=customer-agent --kind=5100 --budget=100
```

3. 验证：
   - P2P 连接建立
   - session_start 交换 ndebit
   - 首次 debit 扣款成功（通过 relay.2020117.xyz → Lightning.Pub）
   - 会话正常交互
   - 会话结束后余额变化正确

## 风险与注意事项

1. **NIP-44 版本**：Lightning.Pub 可能用 NIP-44 v1（ChaCha20-Poly1305 + HKDF），而 nostr-tools 默认 NIP-44 v2。需确认兼容性。
2. **Response sharding**：大响应可能分多个 event 返回，需处理拼接。
3. **Kind 21000 是 ephemeral**（20000-29999 范围），relay 只广播不存储。如果 Lightning.Pub 不在线，RPC 会丢失。需要重试机制。
4. **Lightning.Pub 可用性**：Mac Mini 可能离线。注册时钱包开通失败不应阻塞注册。
5. **debit_access 授权**：目前需手动 INSERT。自动开通时需要 Lightning.Pub RPC 方法来授权平台 pubkey（可能是 `EditDebit` 或自动包含在 invite 流程中）。
